# networking testdata — provenance

Fixtures under `gossipsub.json` and `wire-parity/` are verbatim copies from the external
reference implementation. Fixtures under `synthetic/` are **not** — they are generated by this
workspace's own encoder. Each section below states which it is; read the section before treating
any vector as cross-implementation evidence.

## `gossipsub.json`

Carries 4 cases covering the gossipsub message-id input/output triplet:

- `topic_hex`, `raw_data_hex`, `snappy_mode`, optional `decompressed_hex`.
- `domain_hex`, `hash_input_hex`, `message_id_hex`.

Cases cover the three snappy-mode branches (`none`, `success`, `failure`) plus a binary-payload case. Every field is verified byte-for-byte by the co-located parity test in [src/gossipsub.rs](../../src/gossipsub.rs):

1. `build_message_id_hash_input(domain, topic, payload)` matches `hash_input_hex`.
2. `compute_gossipsub_message_id(domain, topic, payload)` matches `message_id_hex`.

This is the strongest cross-implementation parity guarantee we can land before the libp2p adapter exists — the layout and the SHA-256 truncation are byte-identical to the external reference for every documented case.

## `wire-parity/*.ssz`

Verbatim SSZ-encoded container blobs from the reference implementation. The source repository
pairs each with a `.root.hex` HashTreeRoot file; those roots are not copied here yet, so these
fixtures currently assert byte-parity only. Copying them would give the upstream containers an
independent root anchor — worth doing, tracked separately.

| Fixture                              | Container         |
| ------------------------------------ | ----------------- |
| `empty.blockbody.ssz`                | `BlockBody`       |
| `genesis-4val.state.ssz`             | `State`           |
| `genesis-anchor.checkpoint.ssz`      | `Checkpoint`      |
| `slot12-justified.checkpoint.ssz`    | `Checkpoint`      |
| `slot1.blockheader.ssz`              | `BlockHeader`     |
| `slot1-empty.block.ssz`              | `Block`           |
| `slot7.attestationdata.ssz`          | `AttestationData` |

`slot7.attestationdata.ssz` was renamed from `slot7-vote.vote.ssz`; its bytes are unchanged.
`AttestationData` is a pure rename of the retired `Vote` container — identical field order,
identical 128-byte layout, identical root — so the upstream blob stays valid and keeps its
cross-implementation value.

## `synthetic/*.ssz`

**Generated by this workspace's own encoder — NOT copied from the reference implementation.**

These vectors witness codec self-consistency and the snappy + length-prefixed framing path over
a realistic multi-KB attestation payload. They are **not** cross-implementation parity evidence
and must not be read as such.

| Fixture                            | Container                    | Size | HashTreeRoot                                                       |
| ---------------------------------- | ---------------------------- | ---- | ------------------------------------------------------------------ |
| `validator3.signedattestation.ssz` | `SignedAttestation`          | 3252 | `f698770b0bf6ae48b597bee138698b4829b5452d762f4ba9b2db56a32c18fbeb` |
| `two-attestations.blockbody.ssz`   | `BlockBody`                  |  276 | `0a786852dc25250a5f62918d10bc7a2d19d448cd4b696f015d2ca3ad8942fe10` |
| `slot1-empty.signedblock.ssz`      | `SignedBlockWithAttestation` |  236 | `6210c7d3a20a8d046283fdbd2257543c3ee100f29342fa4c48d9095d19dfbf50` |

The block-envelope refactor moves `slot1-empty.signedblock.ssz` here (from `wire-parity/`) as a
self-generated devnet-1 vector, and re-shapes `two-attestations.blockbody.ssz`: the block body
now holds plain `Attestation` elements (2 × 136 + 4 offset = 276 bytes), with per-vote signatures
carried by the block-signature list on `SignedBlockWithAttestation` rather than inside the body.

They exist because the devnet-1 attestation wire has no published upstream corpus yet. The
devnet-0 `SignedVote` blobs they replace (4136 and 8276 bytes) could not be re-encoded into the
new shape — there was nothing to regenerate *from* — so self-generating them in place would have
left our own encoder's output sitting under the "verbatim copies" banner above, proving only
`decode(encode(x)) == x` while reading as cross-client evidence.

Regenerate with [examples/regen_synthetic.rs](../../examples/regen_synthetic.rs):

```
cargo run -p lean-wire --example regen_synthetic
```

Commit the emitted bytes and copy the printed roots into the table above. When a live devnet-1
peer supplies real bytes, these move back under `wire-parity/` and this section shrinks.

The roots above are asserted by `synthetic_vector_roots_are_pinned` in
[tests/parity.rs](../parity.rs). They cannot prove the shape is *right* — only this workspace
produced them — but they lock it, so an accidental change to field order or merkleization width
fails a test instead of passing silently.

## Parity test coverage

The parity test in [tests/parity.rs](../parity.rs) validates, for every fixture in both
`wire-parity/` and `synthetic/`:

1. **SSZ byte-parity** — `decode::<T>(fixture)` then `encode(&value)` reproduces the fixture bytes exactly.
2. **Snappy framed wire round-trip** — `encode_req_resp_wire(fixture)` then `decode_req_resp_wire(wire)` returns the original SSZ.
3. **Length-prefixed stream round-trip** — `write_req_resp_frame` then `read_req_resp_frame` returns the original SSZ and leaves the cursor at clean EOF.

For `wire-parity/` these three assert agreement with the reference implementation. For
`synthetic/` they assert only self-consistency plus the pinned roots above.
